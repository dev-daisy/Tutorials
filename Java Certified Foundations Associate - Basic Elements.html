In this video, I'll present some of the basic elements of Java including keywords, comments, variables, identifiers, and data types. Object-oriented programming is a methodology based on defining objects that contain properties and methods. A Java object is a stateful entity with associated behaviors and it's meant to represent a physical or conceptual thing such as the cat, a chair, or a list. The class is an entity blueprint to instantiate objects and can be conceptually extended via sub classes to create further blueprints. And objects methods are the actions that can be performed on the instance and are defined by the class blueprint.

Attributes are represented as variables within a class and instance variables of the instantiated object. These can represent data type such as int, long, float, Boolean, enumerations or enums, and even other classes. Java has syntax rules that a program must follow. Every Java program requires a main() method. This is the entry point into the program where the Java Runtime Environment will start executing. And it has a particular syntax which I'll demonstrate in the video titled Programming Basic Java.

All identifiers in Java are case sensitive. This applies to class names, attributes, methods, and any other identifier types and keywords. Classes in Java are named. The first letter of a class name is typically in uppercase, subsequent words in a class name should be capitalized as well. For example, a class called My Form, the M in My and the F in Form are capitalized. There are some conventions for Java basic syntax for file and method names as follows.

Java class names should match the file name it's contained in, and it's done in a case sensitive way. This is not a strict rule, but it's an expected convention you should follow. Method names within a class should begin with a lowercase letter. Any additional words should start with an uppercase letter. This is known as camel case. If you have a method name called getName, the g in get is lowercase but the N in Name would be uppercase. Java consists of several constructs that combined to form the basics of the Java programming language.

Some of the constructs are as follows. Since Java 5.0, we've had enumerated types which can be used to provide a list of predefined values. These are useful to restrict the valid choices for a value that can be chosen. We have arrays that store multiple values of the same type and is stored on the heap. The values in the array can be accessed by an index value which begins counting from 0. The values can also be accessed using a for-each loop.

Keywords in Java are special reserved words that are part of the Java syntax. Because they are reserved words, class, method, and variable names cannot be a reserved word. Another construct is identifiers. Names used for methods, variables, and classes are called identifiers. Identifiers must start with either an alphabetic character, an underscore character, or the dollar sign character. And identifiers cannot be keywords and they cannot begin with a number, although they can contain numbers. Java contains both access and non-access modifiers.

Access modifiers indicate which classes have access to the class, method, or variable. Non-access modifiers include final, abstract, and strictFP. Final means the class cannot be inherited from. Abstract means the class cannot be instantiated into an object. And strictFP restricts floating point calculations. And variables in Java can be a local variable, a static class variable, or a non-static instance variable. And that concludes this presentation on Java basic elements.

  • describe the conventions that should be followed when writing Java programs in regards to formatting, naming conventions, programming practices, and Javadoc
[Video description begins] Topic title: Java Style Guide. Your host for this session is Steve Scott. [Video description ends]

In this video, I'll describe the conventions that should be followed when writing Java programs in regards to formatting, naming and programming practices. When writing code in Java, there are several style guides that can be used for writing your code. Considerations need to be made when selecting the style guide that works best for you. So have a look at Sun Java style guide, the original Sun Java style guide, as well as Google's and Twitter's. These are all publicly available.

Creating consistent code is crucial when working in large teams or external teams. If you stray too far from the Java norms built up over the past quarter of a century, it just makes it on boarding harder. Because you're not only teaching new developers the details of a code base, but you're also wrangling a non-standard style at the same time. The build tools you use may determine the style guide you use since the tools such as Maven have plugins to auto format code. As you write, tools like IntelliJ make formatting changes automatically, such as indentations, function blocks and Java doc comment blocks.

Java file names must be the same as the public class name in the file so the JVM knows its entry point. If an interface is being defined, then the file name must match the name of the interface. Common file names and class names and interface names are usually nouns or noun phrases. When you're creating a class, the class is the object and the methods are the verbs. Stick to camel casing for fields, methods and variable names.

Java source file names must have the .java for the source file, which get turned into .class for the compiled byte code. And .jar for the Java archive files. There are typical conventions that are followed when organizing Java files. Source files should start with a comment block, the package name, and import statements and then followed by any class or interface declarations. The initial comment block should contain information on the name of the class, the version, the date, and any copyright or licensed information, if necessary. The statements for creating a package or importing a package, class or interface should appear next.

Classes and interfaces should be created with the class or interface statements appearing first, followed by the class, static variables, instance variables, constructors and methods. When indenting and writing a line of code, there are some conventions that should be followed. Indentation should be set to four spaces and a single line of code or comment should be less than 80 characters wide so that the line of code is visible on a screen of any common display size. So 80 is your minimum.

When a line needs to be wrapped, certain considerations need to be made as well. The line break should be done before and operator and after a comma. The second line of code should align in terms of indentation with the previous line of code that had the line break. Java has several methods for adding comments to the code. Some of the methods for adding comments are as follows. We have block comments, which are surrounded by slash star and star slash at the end of the block and are used to add descriptions before methods, algorithms, and data structures.

A block line should be added before the block comment. Single line comments are indented at the same level as the code they proceed. They're useful for adding information about a line of code, but don't overdo this style and use it sparingly. Trailing comments are comments that appear on the same line as the code and are usually short and typically use block comment style. End-of-line comments are similar to trailing comments. The main difference is the syntax used for end-of-line comments, which are the two forward slash comments.

Some comments follow the Java doc convention and add special symbols to the comment blocks so that they can be processed by the Java doc tool to generate documentation. Flow control statements in Java, such as loops and exception handling, follow a specific syntax based on the type of statement. Across all of these statements, the code blocks belonging to the statement, called its body, are within braces. The for statement is a loop that starts with the for keyword, followed by the initialization condition, an update or increment section, with each section separated by a semi-colon and all enclosed in parentheses.

The while statement is a loop that starts with the while keyword with a single condition containing a Boolean condition on whether to continue looping. The do-while statement is a loop that begins with the do keyword, followed immediately by the loop body and braces. And just outside of the closing brace is the while keyword, with its condition enclosed in parentheses. The switch statement is the flow control statement, which starts with the switch keyword, followed by the condition in parentheses, followed by zero or more case keywords and the default keyword.

Code blocks after the case and default statements end with a break statement unless you want the program to fall through to the next case or default section. One of the more common errors when writing switch statements is forgetting the break keyword to end a case. Then the try-catch statement is Java's flow control for exception handling, which starts with the try keyword, followed by a try block, enclosed in braces, followed by a catch section and, optionally, a finally section can be added if necessary. The catch keyword is followed by the exception class to be caught in parentheses.

Statements in Java should follow specific conventions when being written based on the type of statement being written. Each line of code should contain a simple statement. Compound statements are enclosed in braces and should be indented one level past the brace. Braces should be used when they're part of a control statements, such as an if-then-else statement. Return statements to return control to the calling method of a function does not need to enclose the return value in parentheses.

If you use parentheses as part of a return, it should only be for clarity in the return value itself. Otherwise, parentheses should not be used. If-then-else statements have blocks with enclosing braces for each section. A block that contains only one line of code can leave out the braces, but some coding conventions recommend not leaving out the braces ever, even for one-line blocks. And that concludes this presentation on Java style guide.
