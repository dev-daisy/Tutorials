2. Video: Privileges (it_dfosfd_16_enus_02)

In this video, learn how to differentiate between system and object privileges.

Objectives

  • differentiate between system and object privileges

[Video description begins] Topic title: Privileges. The presenter is Aaron Sampson. [Video description ends]

In this presentation, we'll differentiate between the two types of privileges available in Oracle. System privileges and object privileges. Now, the system privileges allow you to perform a particular action within the database but they typically aren't concerned with a specific object. Of course, this is the object privilege, and it allows you to manipulate the content of the database objects. So objects such as tables, views, sequences, really any other object can be managed by assigning object privileges.

[Video description begins] A diagram is displayed, which shows that the database security is of two types: system security and data security. System Privileges. [Video description ends]

Now taking a closer look at the system privileges, for starters, there are a lot of them. More than 200 privileges are available. But they are typically held by the DBA, whoever is the database administrator. They have higher level system privileges for various tasks such as, creating new users, removing users, granting other privileges to other users. And perhaps backing up tables or several different databases. So again, these are effectively administrative tasks.

They typically don't operate against a specific object like a table or a view. So in most cases you will certainly have both depending on what your job is. But the degree of privileges that you have is certainly very controllable. The system administrator can determine who is able to hold which privileges and it can be any combination of object privileges and/or system privileges. So of course, we'll see some examples as to how to assign these privileges. But as mentioned, almost every user will have a certain level of privilege in either one or the other or both.

4. Video: Roles (it_dfosfd_16_enus_04)

Upon completion of this video, you will be able to describe the purpose of a role.

Objectives

  • describe the purpose of a role

[Video description begins] Topic title: Roles. The presenter is Aaron Sampson. [Video description ends]

In this presentation, we'll introduce the concept of a role. And quite simply, a role is a named collection of privileges that generally are related. And this makes it much easier to grant and/or revoke, or just generally maintain users and their privileges. Now, you certainly can operate with individual users and there are always instances where you may still do this. But if a number of users need the same level of access to effectively the same set of objects, then it makes much more sense to use a role.

[Video description begins] What is a Role? [Video description ends]

If you have even just a few users, such as three users accessing four different objects and every user needs to access every object, then you have 12 connections to make, if you will.

[Video description begins] In the diagram displayed, the users are allocated the privileges without a role. [Video description ends]

User one has access to four objects. User two has access to four more or the same four, I should say. And user three has access to the same four. So you're managing 12 different sets of privileges. But if you use a role, you can then assign the privileges to the role and this dramatically reduces the amount of connections, for lack of a better word.

[Video description begins] In the diagram displayed, the users are assigned the Manager's role. The privileges are allocated with a role. [Video description ends]

The role can have access to the four objects, that's it. Then you simply populate the role. Now, this is hardly a new approach, almost every type of IT administration makes use of users and groups. And they all essentially perform the same task. Rather than managing a lot of individuals with access to a lot of resources, let's just group them together. So that all of the individual users who require that same level of access are just placed within the role or the group. So it's just meant to simplify the approach to handling all of your users with respect to the objects that they are able to access.
