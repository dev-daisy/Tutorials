In this video, you will identify the different types of constraints and how these are used to enforce rules at a table level.

Objectives

[Video description begins] Topic title: Overview of Constraints. The presenter is Aaron Sampson. [Video description ends]

In this presentation we'll take a look at including constraints in your tables, which are designed to enforce rules at the table level. To help ensure consistency and the overall integrity of the database. So as a very simple example, if you're allowing people to enter values into a particular column that was maybe just character based. Let's go with the name of a city. If there were no constraints on that column, then one user might enter the city using its full name, such as New York. But then another user might enter the same value but using abbreviations, they might put in NY. Then another user might come along and put in NYC. Then another user might enter N, period, Y, period.

So you can see that in that simple example, we have inconsistent data. If I then were to execute a query that specified to return all customers in New York, and then I spelled out the name. Well, I would only get that one record back, where the user actually entered in New York. All of the others would not be returned. So clearly that represents a huge problem in the design of the database. You want to ensure consistency and integrity so that everyone enters the same value. So there are several different constraint types that can be implemented. Not null, which is just a property that you can set on a column, is a constraint. It forces you to enter something, you cannot leave it blank. A unique constraint prevents duplicates. A primary key essentially does that as well. But, in fact, it would be really a combination of not null and unique.

Because a primary key does not allow a null value and it has to be unique. A foreign key represents a primary key value from some other table. But it ensures that the value exists. So if I'm creating an order, and it belongs to a particular customer, I have to use a valid customer ID, for example, to indicate the owner of this order. I can't just create an order or a customer that does not exist. So, again, that is a constraint, I can't just make up a value. And a check constraint is essentially any kind of constraint that you design yourself. So, for example, the city issue that I just mentioned. Not allowing users to enter New York, NY, N, period, Y, period, that would be an example of a check constraint. And they typically represent just your own business rules in many cases.

[Video description begins] Constraint Guidelines. [Video description ends]

Now, there are some guidelines. You can name a constraint yourself or the Oracle server can generate an automatic name by using the format of SYS_C for constraint and then n is just a generated number that is just randomly selected. So typically you would want to name them yourself so you know what you're working with. And you can create a constraint at either the time of table creation or afterward, it really doesn't matter, it's up to you. You can define a constraint at the column or the table level. Now, each individual constraint will operate on a specific column.

But it really is considered more so a table-level object because if you consider the primary key, although that generally represents a single column, it actually could include more than one column. Plus it helps to ensure that every record in the table is unique. So it's more so a table-level object. Plus any single constraint can operate on multiple columns. So typically they are considered to be table-level objects, but you can define it on a specific column. And you can view a constraint in the data dictionary to simply see its definition, its properties. So here we see some examples of defining constraints. And, again, you can do it during the creation of a table itself. But it can be done afterward as well.
